# 유저뷰 기능기획 02 — 매칭 만족도 피드백

> 버전: v1.0 | 작성일: 2026.07.08 | 상태: 📋 초안
> 우선순위: **2순위** (Sol 논의 신규항목 6건 중) — 없으면 파일럿에서 매칭 로직을 튜닝할 데이터가 생기지 않음
> 근거: 화면4 §4(concept_type × axis 친화도 "v1 초기값 — 파일럿 데이터로 튜닝"), SSOT v1.2 §4(세그먼트 산출조건 "v1 초안 — 파일럿 데이터로 튜닝"), 화면6 §3-⑤(미니 리포트)
> 성격: **학습 루프(learning loop)** — 관람객의 반응을 수집해 매칭·세그먼트 로직을 개선하는 장치

---

## 1. 왜 필요한가

현재 서비스의 핵심 로직 두 개가 **전부 추정값**이다.

| 로직 | 현재 상태 | 문서 |
|---|---|---|
| Bridge 매칭 (concept_type × axis 친화도 테이블) | "v1 초기값 — 파일럿 데이터로 튜닝" | 화면4 §4 |
| 세그먼트 산출 조건 (4종 분류 규칙) | "v1 초안 — 파일럿 데이터로 튜닝" | SSOT §4-1 |

둘 다 "파일럿 데이터로 튜닝한다"고 적혀 있지만, **정작 그 데이터를 수집하는 장치가 없다.** 지금 상태로 파일럿을 돌리면 "몇 명이 팝업에 갔는가"(전환율)는 알 수 있어도, **"왜 안 갔는가 / 추천이 맞았는가"는 영원히 알 수 없다.**

MVP 성공지표 3번(v3.0 §9: *"수집된 취향 데이터가 실제 굿즈 구매 의사와 일치하는가"*)은 사실상 이 기능 없이는 측정 불가능하다.

---

## 2. 설계 원칙 — "가볍게, 자연스러운 순간에"

전시 관람은 감성적 경험이다. 설문 폼을 들이밀면 여운이 깨진다.

- **최대 1탭.** 별점·이모지 수준. 서술형은 선택.
- **가치를 받은 직후에만 묻는다.** 리포트를 받은 순간, 팝업 체험을 마친 순간.
- **거절 가능.** 스킵해도 아무 불이익 없고, 재차 묻지 않는다.
- **묻는 대상은 "AI 추천"이지 "전시"가 아니다.** 전시 만족도는 기획사 소관, 우리가 학습해야 할 건 매칭 정확도.

---

## 3. 수집 지점 3곳

### F1. 초대장 반응 (화면4 — 리포트 직후)

가장 중요한 지점. **추천을 받은 순간의 직감**을 잡는다.

```
┌──────────────────────────────┐
│ 💌 당신을 위한 제안 3          │
│ ┌──────────────────────────┐ │
│ │ 1  어비스 티 라운지         │ │
│ │    [보러가기 →]            │ │
│ │    이 제안, 어떠세요?       │ │  ← 카드 하단
│ │    [👍] [👎]              │ │     1탭, rank별 개별 수집
│ └──────────────────────────┘ │
└──────────────────────────────┘
```

- **rank 1 카드에만 노출**(전체 카드에 붙이면 부담). 다중 스핀오프에서도 rank 1의 정확도가 핵심 지표.
- 👎 선택 시에만 **선택적 사유 칩** 1회 노출: `취향과 안 맞아요` / `너무 멀어요` / `관심 없는 형태예요` / `가격이 부담돼요`
- 이 데이터로 **concept_type × axis 친화도 테이블을 직접 보정**할 수 있다 (어떤 axis 조합에서 어떤 concept_type이 실패하는가).

### F2. 팝업 체험 후 (화면6 — 미니 리포트)

**추천이 실제 경험과 맞았는지**를 검증한다. 화면6 §3-⑤ 미니 리포트에 통합.

```
오늘의 여운 마무리
─────────────────
'어비스 티 라운지'는 어떠셨나요?
   ⭐⭐⭐⭐☆        ← 5점 척도, 1탭
─────────────────
[차기 전시 소식 받기]
```

- 별점만 필수, 서술형 한 줄은 선택(placeholder: "가장 좋았던 순간은?").
- 이 값이 F1의 👍/👎와 **짝을 이룰 때 가장 유용**하다: "👍 하고 갔는데 별점 2점" = 매칭은 맞았지만 콘텐츠 품질 문제 / "👎 였는데 별점 5점" = 매칭 로직이 사용자 취향을 과소평가.

### F3. 감정 프로필 확인 (화면4 — 페르소나 카드)

세그먼트 산출 규칙 튜닝용. **간접 신호**라 부담이 없다.

```
[페르소나 카드]
"Deep Ocean Dreamer"
─────────────────
이 표현이 나와 닮았나요?  [예] [글쎄요]
```

- `TASTE_PROFILE.segment_tag` 정확도의 프록시 지표.
- "글쎄요" 응답이 특정 세그먼트에 몰리면 산출 조건(§4-1)이 잘못된 것.

## 4. 데이터 계약 (SSOT 개정)

```
MATCH_FEEDBACK               🆕 신규 엔티티
  feedback_id (PK)
  exhibition_id (FK), session_id (FK)
  invite_id (FK)?              F1일 때
  type: enum                   invite_reaction | popup_rating | persona_accuracy
  value: string                F1: up|down / F2: 1~5 / F3: yes|unsure
  reason: enum?                F1 👎 시 사유 칩
  comment: string?             F2 서술형(선택)
  created_at

SPINOFF_INVITE
  (기존 rank, match_score 활용 — 피드백과 조인해 스코어 정확도 역산)
```

### 신규 이벤트

| 이벤트 | 시점 |
|---|---|
| `feedback.invite_reaction` | F1 👍/👎 |
| `feedback.popup_rating` | F2 별점 |
| `feedback.persona_accuracy` | F3 응답 |

전부 **퍼널 외 지표**. 전환율 집계에 영향 없음.

---

## 5. 학습 루프 — 수집한 데이터를 어떻게 쓰는가

수집만 하고 안 쓰면 의미 없다. **파일럿 종료 후 튜닝 절차**를 미리 정의한다.

| 데이터 | 개선 대상 | 방법 |
|---|---|---|
| F1 👍/👎 + `match_score` | concept_type × axis 친화도 테이블 (화면4 §4) | axis 조합별 👍률 집계 → 낮은 셀의 가중치 하향 |
| F1 사유 칩 `너무 멀어요` | 위치기반 추천 필요성 판단 (신규항목 6번) | 이 사유 비중이 높으면 위치 요소 도입 근거 확보 |
| F2 별점 vs F1 반응 | 매칭 오류 vs 콘텐츠 품질 문제 구분 | 위 §3-F2의 2×2 매트릭스 |
| F3 정확도 | 세그먼트 산출 조건 (SSOT §4-1) | 세그먼트별 "글쎄요" 비율 → 임계값 조정 |

**어드민 연동**: Insight Reporter(v3.0 §2)의 리포트에 위 지표를 포함 → 기획사에게도 "우리 추천이 얼마나 맞았는가"를 보여줄 수 있다. B2B 신뢰 자산.

---

## 6. MVP IN / OUT

| IN | OUT (v2+) |
|---|---|
| F1 초대장 👍/👎 + 사유 칩 | 서술형 심층 피드백 폼 |
| F2 팝업 별점 5점 척도 | NPS·재방문의향 등 추가 설문 |
| F3 페르소나 정확도 2지선다 | 실시간 매칭 로직 자동 보정(온라인 학습) |
| 파일럿 종료 후 수동 튜닝 | A/B 테스트 프레임워크 |

## 7. 남은 결정 포인트

1. **F3(페르소나 정확도)을 MVP에 넣을지** — 3곳 모두 물으면 피드백 피로가 쌓일 수 있음. (제안: F1·F2 우선, F3은 리포트 하단에 아주 작게 — 스킵률 높아도 무방)
2. **F2 별점의 낮은 점수 처리** — 1~2점 응답 시 개선 요청 창구를 열지, 그냥 수집만 할지. (제안: 파일럿에선 수집만. CS 리소스 없음)
3. **피드백 인센티브** — 응답률이 낮으면 데이터가 안 쌓임. 소액 혜택(굿즈 할인 등) 연결 검토 필요 여부.

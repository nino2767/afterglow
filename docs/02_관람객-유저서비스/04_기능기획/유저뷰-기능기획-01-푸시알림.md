# 유저뷰 기능기획 01 — 푸시 알림

> 버전: v1.0 | 작성일: 2026.07.08 | 상태: 📋 초안
> 우선순위: **1순위** (Sol 논의 신규항목 6건 중) — MVP 시나리오 8(오픈 대기→재방문)이 현재 이 기능 부재로 성립 불가
> 근거: SSOT v1.2, 화면4 §5(오픈 전 opt-in), 화면5 §4(D2 카운트다운), 화면10 우선순위 엔진, 스핀오프 운영정책 상태머신
> 성격: **횡단 기능(cross-cutting)** — 특정 화면이 아니라 화면4·5·7·10 + 서버 스케줄러에 걸쳐 있음

---

## 1. 왜 필요한가

현재 스펙은 관람객에게 **"오픈하면 알려드릴게요"라고 약속만 하고 지킬 수단이 없다.**

- 화면4 §5: 팝업 미운영 기간 → 초대장 대신 "오픈 알림 받기" opt-in
- 화면5 §4(D2): 팝업 오픈 전 접근 → 카운트다운 + 오픈 알림 opt-in
- 화면10: 팝업이 `ready`→`open_concurrent`로 전이돼도, 관람객이 **스스로 앱을 다시 열어야만** 알 수 있음

→ 시나리오 8(준영: 오픈 전 알림 신청 → 2주 뒤 팝업 오픈 → 재방문)이 **운에 의존**하게 됨. 전시 IP의 수익 수명 연장(v3.0 핵심 가치)이 알림 없이는 반쪽.

---

## 2. 채널 선택

| 채널 | 장점 | 단점 | 판단 |
|---|---|---|---|
| **웹 푸시(Web Push)** | 앱 설치 불필요, 무료, 웹앱 기조와 일치 | iOS Safari는 홈화면 추가(PWA) 필수 — 실사용 도달률 낮음 | 보조 |
| **카카오 알림톡** | 국내 도달률 최상, 관람객 대부분 보유 | 발송 비용, 템플릿 사전 승인 필요, 전화번호 수집 필요 | **주** |
| 이메일 | 저비용 | 즉시성 낮음, 개봉률 낮음 | 보조 |

**결정: 카카오 알림톡 우선 + 이메일 폴백.**

근거 — 관람객 계정이 이미 카카오 소셜 로그인 기반(SSOT §7-1)이고, 팝업 오픈·종료임박 같은 **시의성 있는 알림**은 도달률이 곧 기능 가치. 웹 푸시는 iOS 제약으로 MVP 파일럿(국내, 모바일 웹) 환경에서 신뢰할 수 없음.

> ⚠️ 카카오 알림톡은 **전화번호**가 필요하다. 현재 SSOT §7-2의 ACCOUNT에는 전화번호 필드가 없음 → §5 참조.

## 3. 알림 유형 (4종)

전부 팝업 상태머신(운영정책 §1)에 연동. **알림은 이벤트가 아니라 상태 전이의 결과**로 발송된다.

| # | 유형 | 트리거 | 대상 | 문구 예시 |
|---|---|---|---|---|
| N1 | **팝업 오픈** | `ready` → `open_concurrent` 전이 | 해당 본전시에서 오픈알림 opt-in한 관람객 | "S님이 기다리던 '어비스 티 라운지'가 열렸어요" |
| N2 | **종료 임박** | `open_solo` 상태에서 종료 D-3일 도달 | 초대장 보유 + 미체크인(`redeemed` 없음) 관람객 | "그 푸른 빛, 3일 뒤면 사라져요" |
| N3 | **초대장 발급** | `invite.issued` 직후 (선택 발송) | 관람 완료자 | "오늘의 여운을 정리해뒀어요" |
| N4 | **차기 전시** | 신규 전시 오픈 시 | 차기전시 opt-in 관람객 | (전시별 카피) |

### 발송 규칙

- **N2는 미체크인자에게만** — 이미 방문한 사람에게 "곧 종료" 알림은 무의미하고 스팸으로 인식됨. `invite.redeemed` 여부로 필터.
- **일 1건 상한** — 여러 스핀오프 초대장을 가진 관람객에게 N2가 동시에 여러 건 걸릴 수 있음(다중 스핀오프 확정으로 발생). 이 경우 **우선순위 엔진(화면10 §A)과 동일 규칙**으로 1건만 발송.
- **동일 유형 재발송 금지** — N1은 팝업당 1회, N2는 초대장당 1회.
- 발송 시간대: 09:00~21:00 (야간 발송 차단).

## 4. 관람객 접점 (화면 반영)

### 4-1. opt-in 수집 (기존 화면 보강)

| 화면 | 기존 | 추가 |
|---|---|---|
| 화면4 (오픈 전) | "오픈 알림 받기" 카드 | 탭 시 로그인 유도(비로그인) → 알림 수신 채널 확인 |
| 화면5 (D2 오픈 전) | 카운트다운 + opt-in | 동일 |
| 화면5 (D1 만료 후) | 차기전시 opt-in | 동일 |

### 4-2. 알림 설정 (화면7 마이페이지 — 기존 §2-2 구체화)

```
🔔 알림 설정
  전체 알림                    [ON/OFF]   ← 마스터 스위치
  ─────────────────────
  팝업 오픈 알림                [ON/OFF]   ← N1
  종료 임박 알림                [ON/OFF]   ← N2
  차기 전시 소식                [ON/OFF]   ← N4
  ─────────────────────
  받는 채널: 카카오 알림톡 (010-****-1234)
  [채널 변경]
```

- N3(초대장 발급)은 트랜잭션성 알림이라 토글 대상에서 제외(마스터 스위치로만 제어).
- **수신거부는 알림 하단 링크로도 1탭 가능**해야 함(법적 요구).

### 4-3. 알림 → 앱 진입 (딥링크)

| 알림 | 도착 화면 |
|---|---|
| N1 팝업 오픈 | 화면5 (해당 invite_id 랜딩) |
| N2 종료 임박 | 화면5 (동일) |
| N3 초대장 발급 | 화면4 (리포트) |
| N4 차기 전시 | 화면10 (홈) |

비로그인 상태로 딥링크 진입 시: 게스트로도 열람 가능(SSOT §7-1 게스트 우선 원칙 유지).

## 5. 데이터 계약 개정 (SSOT)

```
ACCOUNT
  + phone: string?              🆕 카카오 알림톡 발송용. 카카오 로그인 시 동의 항목으로 수집
  + notify_channel: enum        🆕 kakao | email  (기본: phone 있으면 kakao)

NOTIFICATION_SUBSCRIPTION      🆕 신규 엔티티
  subscription_id (PK)
  account_id (FK)
  exhibition_id (FK)            어느 전시/팝업에 대한 구독인지
  type: enum                    popup_open | closing_soon | next_exhibition
  active: bool
  created_at

NOTIFICATION_LOG               🆕 신규 엔티티 (중복 발송 방지 · 성과 측정)
  log_id (PK)
  account_id (FK)
  type, exhibition_id, invite_id?
  sent_at, opened_at?           opened_at = 딥링크 진입 시각
```

### 신규 이벤트

| 이벤트 | 시점 |
|---|---|
| `notification.subscribed` | opt-in 시 |
| `notification.sent` | 발송 성공 |
| `notification.opened` | 딥링크 클릭 |
| `notification.unsubscribed` | 수신거부 |

**퍼널 오염 방지**: 알림 이벤트는 전환 퍼널 6단계와 **별도 집계**. 단, `notification.opened` → `invite.landing_viewed`로 이어지면 그건 정상적인 4단계 카운트(유입 채널만 알림일 뿐).

## 6. 서버 사이드 요구사항

- **스케줄러 필요**: N1은 상태 전이 시점(이벤트 기반), N2는 매일 배치로 D-3 도달 여부 스캔.
- 상태 전이 자동화는 이미 운영정책 §1에 정의됨(`main.end` 도달 시 자동 전이) — 그 전이 훅에 알림 발송을 연결.
- 시나리오 인벤토리 H1(21일 상한 자동 전이)과 동일 메커니즘 재사용.

## 7. MVP IN / OUT

| IN | OUT (v2+) |
|---|---|
| 카카오 알림톡 N1·N2 | 웹 푸시(PWA) |
| 마이페이지 알림 토글 3종 | 알림 시간대 개인 설정 |
| 딥링크 진입 | 리치 템플릿(이미지·버튼 다수) |
| 이메일 폴백 | N3 초대장 발급 알림(선택 — 스팸 우려로 파일럿에선 OFF 권장) |

## 8. 남은 결정 포인트

1. **전화번호 수집 시점** — 카카오 로그인 시 동의항목에 포함할지, 알림 신청 시점에 별도로 받을지. (제안: 알림 신청 시점 — 로그인 마찰 최소화)
2. **N3(초대장 발급 알림) 파일럿 발송 여부** — 관람 직후 앱을 보고 있는데 알림까지 보내면 과할 수 있음. (제안: 파일럿에선 OFF)
3. **알림톡 발송 비용 부담 주체** — AFTERGLOW vs 기획사(운영비 포함). 비즈니스 구조(v3.0 §3)와 연결됨.

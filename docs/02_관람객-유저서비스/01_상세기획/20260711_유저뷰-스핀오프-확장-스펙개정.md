> 근거: Sol 논의 메모(노션 "스핀오프 상세화"), SSOT v1.2, 유저뷰 IA, 화면4·5·6 상세기획

## 1. 다중 스핀오프 (1 main : N spinoff 활성화)

### 1-1. 배경 · 결정

- SSOT v1.1에서 구조상 `1 main : N spinoff`를 허용했으나 **운영은 1:1로 시작**하기로 보류했던 항목.
- Sol 논의 결과: **본전시별로 운영 가능한 스핀오프 개수를 운영자가 설정**하고, 관람객에게는 취향 매칭 상위 N개를 모두 제안한다.
- 개수 결정 주체 = **운영자(어드민)**. 관람객은 그중 취향 매칭 결과를 받는다.

### 1-2. SSOT 개정 사항

```
EXHIBITION (type=main)
  + max_spinoff_recommendations: int   🆕 기본값 1. 이 본전시에서 관람객 1인에게
                                        제안할 스핀오프 최대 개수 (운영자 설정)

SPINOFF_INVITE
  + rank: int                          🆕 추천 순위 (1=최상위). 동일 세션 내 복수 발급 시 부여
  + match_score: float                 🆕 concept_type × axis 친화도 스코어 (정렬·튜닝 근거)
```

- 한 세션(`session_id`)에서 **복수의 SPINOFF_INVITE 발급 가능** — 기존 1건 전제 해제.
- `invite.issued` 이벤트는 **발급된 초대장 건수만큼** 발화하되, 퍼널 3단계 카운트는 **세션당 1회**로 집계(중복 카운트 방지). → 전환율 20% 지표의 분모/분자 정의가 흔들리지 않음.

### 1-3. Bridge 매칭 로직 개정 (기존 화면4 §4)

```
기존: concept_type별 axis 친화도 스코어링 → 최고점 1종 선택
개정: concept_type별 axis 친화도 스코어링 → 상위 N종 선택
      N = main.max_spinoff_recommendations (운영자 설정, 기본 1)
      단, 실제 운영 중인 스핀오프 수가 N보다 적으면 그 수만큼만 발급
```

- 스코어 동점 시: 팝업 종료 임박순(유효기간 짧은 것 우선) → 그래도 동점이면 concept_type enum 정의 순서.
- 매칭 가능한 스핀오프가 0개면: 기존 default 폴백 유지(운영자 지정 1종) — **초대장 미발급 케이스는 여전히 없음**.

### 1-4. 화면4 (여운 리포트) 개정

초대장 섹션이 단수 카드 → **추천 순위대로 전체 나열**.

```
┌──────────────────────────────┐
│ 💌 당신을 위한 제안 3          │  건수 표기
│ ┌──────────────────────────┐ │
│ │ 1  어비스 티 라운지         │ │  rank 1 (가장 큰 카드, 강조)
│ │    [sensory] 매칭 92%      │ │  match_score 노출 (선택)
│ │    personalized_copy       │ │
│ │    [보러가기 →]            │ │
│ └──────────────────────────┘ │
│ ┌──────────────────────────┐ │
│ │ 2  딥블루 레코드            │ │  rank 2 이하는 컴팩트 카드
│ │    [archive] · D-12        │ │
│ └──────────────────────────┘ │
│ ┌──────────────────────────┐ │
│ │ 3  루미너스 랩              │ │
│ └──────────────────────────┘ │
└──────────────────────────────┘
```

- **rank 1은 시각적으로 강조**(큰 카드 + personalized_copy 전문), rank 2 이하는 컴팩트 카드(제목·concept_type·유효기간).
- 사용자가 카드를 골라 탭 → 해당 `invite_id`의 화면5로 진입 (`invite.landing_viewed`는 탭한 초대장 기준 발화).
- `max_spinoff_recommendations = 1`(기본값)이면 기존과 동일한 단일 카드 UI로 자연 축약 — 하위 호환.

### 1-5. 화면5 (스핀오프 랜딩) 영향

- 랜딩 자체는 여전히 **초대장 1건 단위**(`?invite_id=`). 구조 변경 없음.
- 단, 상단에 "다른 제안 보기 →" 링크 추가(화면4 또는 화면9로 복귀). 여러 제안 중 하나를 보다가 다른 걸 비교할 수 있어야 함.

### 1-6. 영향 없는 화면 (이미 복수 전제로 설계됨)

- **화면9(초대장 목록)** — 원래부터 복수 초대장을 그룹·정렬해 보여주는 구조. 변경 없음.
- **화면10(홈 우선순위 엔진)** — "여러 초대장 중 최우선 1건만 노출" 로직이 이미 있음. 스핀오프가 몇 개든 그대로 작동.

### 1-7. 어드민 영향 (별도 반영 필요)

- **ConceptBot**: 본전시 설정에 `max_spinoff_recommendations` 필드 추가 (기본 1).
- **Report**: 퍼널 3단계 집계 시 세션당 1회 규칙 적용 확인.

## 2. 스핀오프 진입 코치마크 (경량 온보딩)

### 2-1. 배경 · 결정

Sol 의견 "스핀오프 진입 시 온보딩 필요" ↔ 기존 화면6 설계 "본전시 관람자는 온보딩 없이 연속 경험"의 절충.

- **데이터 재수집형 온보딩(화면1 방식: 동의·닉네임·키워드)은 넣지 않는다.** 연속성 경험("다시 만났네요")이 AFTERGLOW의 핵심 차별점이며, 이미 계정·취향 데이터를 보유했으므로 재수집은 마찰만 유발.
- 대신 **기능 안내용 경량 코치마크(플로팅 툴팁)**를 도입. 본전시와 UI가 다른 스핀오프 고유 기능(존 지도, 존 QR, 팝업 큐레이터)의 위치만 알려준다.

### 2-2. 사양 (화면6 §3에 추가)

| 항목 | 내용 |
|---|---|
| 노출 시점 | 화면6 최초 진입 시 1회 (`spinoff.session_start` 직후) |
| 형태 | 1~2장 스킵 가능한 플로팅 툴팁. 전체화면 모달 아님(웰컴 첫마디를 가리지 않게) |
| 내용 | ① "여기가 이 공간의 존 지도예요" ② "존 앞에서 QR을 찍거나, 큐레이터에게 물어보세요" |
| 재노출 | 안 함(로컬 저장). 마이페이지에서 "안내 다시 보기" 제공(선택) |
| 대상 | **본전시 관람자·미관람자 공통** |
| 미관람자 추가 | 기존 축약 온보딩(E1 시나리오, 키워드 3개) → 그 다음 코치마크 순서 |

### 2-3. 왜 온보딩이 아니라 코치마크인가 (설계 근거 기록)

Sol 지적의 핵심은 "본전시와 스핀오프의 UI/UX가 달라야 한다"였고, UI가 다르면 사용자가 길을 잃을 수 있다는 우려는 타당하다. 다만 그 해법은 **데이터를 다시 묻는 것이 아니라 새 UI의 사용법을 알려주는 것**이다. 온보딩(데이터 수집)과 코치마크(기능 안내)를 분리함으로써, 연속성 경험을 지키면서 학습 부담을 해결한다.
